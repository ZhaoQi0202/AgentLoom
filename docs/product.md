# AgentCrewChat — 产品规格文档

> **用途**：记录产品构想、功能点、实现索引及已知差距。AI 做开发时优先阅读本文，与 [frontend.md](./frontend.md)、[backend.md](./backend.md) 交叉核对。  
> **维护规则**：对话中新定案或范围变更须直接修改本文对应小节，保持全文自洽。

---

## 1. 产品定位

| 项 | 说明 |
|----|------|
| 形态 | **个人用**桌面客户端（Electron + 本地 Python 后端），非多租户 SaaS |
| 核心体验 | **项目群聊**：多 Agent 像项目组同事一样在时间线里分工、接力，降低「单黑盒助手」感 |
| 平台 | **Windows 优先**，macOS 后续支持 |
| 安装方式 | **自包含安装包**：使用 [python-build-standalone](https://github.com/indygreg/python-build-standalone) 嵌入式 Python 运行时打包进 Electron 安装包（Electron Builder），用户无需自行安装 Python，装完即用 |
| 安装目录 | 首次安装时由用户选择安装目录。所有数据（config/data/workspaces）均在安装目录下，不使用 `%APPDATA%` |

---

## 2. 非目标（现阶段）

- 不提供多租户、云托管、账号体系
- 不为纯视觉牺牲阶段与事件可观测性（群聊须跟得上「现在在干什么」）
- LLM Auto 模式（按阶段自动分配最优模型）— 二期

---

## 3. 项目组空间

- **一个项目组 = 一条大需求的完整生命周期**
- 工作区目录 + 独立 venv 隔离，项目组之间完全独立
- 目录命名：`workspaces/task_<ms>_<slug>/`
- 项目组内产物：当前目录安装的 Skills、生成的脚本、执行结果，全部在自己的工作区目录下
- `meta.json` 存储展示名等元信息

---

## 4. 四阶段工作流

| 阶段 | Agent | 用户感知 | 产物 |
|------|-------|----------|------|
| **1 需求收集** | 晓柔（需求分析师） | 群里多轮对话，引导补充信息 | `requirement.json` |
| **2 架构规划** | 明哲（架构师） | 发方案、等确认 | `blueprint.json` |
| **3 多 Agent 执行** | 小某（执行者） | 多名执行者在群里推进，同层任务并行执行 | `task_outputs/*.json` |
| **4 审核反馈** | 铁口（审核员） | 每任务执行后审核 | 审核结论 |

**主流程**：新建项目组 → 进入群聊 → 阶段 1 → 用户确认 → 阶段 2 → 蓝图 HITL → 阶段 3+4 交织 → 最终汇总

**阶段切换交接规则**：

1. 晓柔完成需求收集后，发一条群消息 @明哲 交接需求摘要（一条消息，不分开发）
2. 明哲思考中显示加载动画，出蓝图后展示方案并等待用户 HITL 确认
3. 用户确认蓝图后，明哲发一条群消息 @每个执行 Agent（小某），分配任务并定义汇报节点
4. 所有执行 Agent 同时入群，按照明哲分配的任务和汇报节点执行
5. 不存在 `hitl_blueprint` 入群消息——HITL 确认是一个交互卡片，不是 Agent 入群

**执行阶段入群规则**：蓝图确认后，所有执行 Agent（小某）同时入群（`agent_join`），不在执行过程中逐个入群。右侧面板一次性展示所有执行 Agent 及其主题色。

**需求收集收口**：模型判断要点齐备后向用户做摘要，用户确认后进入架构阶段并落盘结构化需求，不在未确认时自动切换阶段。

## 5. Agent 角色设定

### 5.1 三大固定 Agent

| Agent | 名字 | 性格 | 主题色 |
|-------|------|------|--------|
| 需求分析师 | **晓柔** | 温和耐心、善于引导，话语亲切，像项目群里贴心的同事 | `#7c3aed`（紫） |
| 架构师 | **明哲** | 严谨理性、言简意赅，说话简洁有力，直接给结论 | `#2563eb`（蓝） |
| 审核员 | **铁口** | 毒舌挑剔但细致，能给执行 Agent 具体指导，直言不讳 | `#ea580c`（橙） |

### 5.2 执行 Agent（动态）

- 名字统一为 **「小某」**，每次从**随机生成的两字中文名**中分配（如：小见、小瑶、小辰），同一项目组内不重复
- 执行 Agent 上限默认 **5 个**，相似工具的任务合并到同一 Agent；用户明确要求时可超过上限
- 性格从**性格池**随机挑选，每次执行风格不同
- 每个执行 Agent 从**执行 Agent 色板**中随机分配一个主题色（不与固定 Agent 主题色重复）；超过 15 个时随机生成 HSL 颜色（`S: 60-80%, L: 40-55%`，避免与已分配色相距过近）

**性格池**：

| 性格 | 表现 |
|------|------|
| 急性子 | 话少、直接给结果、不废话 |
| 碎嘴子 | 喜欢解释过程、说话啰嗦但认真 |
| 自信派 | 语气笃定、偶尔自夸 |

| 乐观派 | 说话积极、喜欢加油打气 |
| 毒舌徒弟 | 受审核员影响、偶尔嘴硬但执行力强 |
| 学院派 | 喜欢引用方法论、说话偏正式 |
| 佛系 | 什么都"好的没问题"、但靠谱 |

### 5.3 消息风格规则

- **性格完全通过提示词实现**，LLM 输出自带风格（方案 A）
- Agent 可在消息中自由使用 emoji，无固定格式限制，体现随机感和活人感
- **禁止裸露 JSON 出现在聊天气泡**，所有面向用户的消息必须是人类可读的自然语言 + Markdown 格式
- **Agent 间交接消息**：LLM 生成拟人化移交语，格式中包含 `@对应名称`，前端将 `@名称` 渲染为对应 Agent 主题色
- **`@名称` 解析规则**：**只匹配已知 Agent 名字列表**（晓柔、明哲、铁口 + 当前项目组的动态执行 Agent 名），不使用通用正则，避免误触发
- **需求收集选项格式**：晓柔给用户选项时使用 A/B/C 格式（如 `A. 方案一` / `B. 方案二`），用户可直接输入 A/B/C 回复选择
- **不存在名为 "experts" 或 "执行者" 的 Agent**：执行阶段没有统一的入口 Agent，每个执行任务由各自的「小某」直接入群工作

## 6. LLM 配置

- 支持**多个 LLM 连接**（OpenAI Compatible、Anthropic），每个连接有名称、API Key、Base URL、模型名
- 支持**连接探测**（测试是否可用）
- **按四阶段分别配置 LLM**：需求收集 / 架构规划 / 执行 / 审核，每阶段独立选用哪个连接
- 执行阶段所有执行 Agent 共用同一个 LLM 配置（一期），后续可细化为按 Agent 类型分配（二期）
- **Auto 模式**（二期）：用户配置可用连接，系统自动为四阶段选最优模型

## 8. 聊天室

### 8.1 消息类型

| 类型 | 说明 |
|------|------|
| `agent_output` | Agent 发言，含头像/名称/Markdown 内容 |
| `agent_thinking` | 思考气泡（打点动画占位） |
| `user_response` | 用户消息 |
| `phase_start` / `agent_join` | 系统消息（阶段切换/Agent 入群） |
| `hitl_interrupt` | HITL 确认卡（蓝图确认等） |
| `hitl_retry_limit` | 审核超限，等待用户决策（附快捷回复按钮） |
| `error` | 错误提示 |

### 8.2 显示规则

- `agent_output` 最小显示 **2 秒**，不足时补足，期间显示思考气泡占位
- 输出未返回时，气泡加载动画先占位
- 同一 Agent 入群提示只渲染一次
- 消息气泡中**禁止裸 JSON**

### 8.3 聊天历史持久化

- **持久化到本地**：切换项目组再切回消息仍在，重启软件也保留
- 按项目组隔离存储

### 8.4 右侧状态面板

- **四阶段分栏**：每阶段一个卡片，显示阶段名称
- 每个阶段卡片内展示对应 Agent 的头像和名称
- 执行阶段按实际拆分的 Agent 数量和名称动态展示
- 每个 Agent 右上角状态点：

| 颜色 | 含义 |
|------|------|
| 灰色 | 未开始 |
| 黄色脉冲 | 进行中 |
| 绿色 | 已完成 |
| 红色 | 有问题 |

## 9. 执行控制

### 9.1 暂停/继续

- **ReAct 迭代级暂停**：后端每个 session 维护 `threading.Event` 信号，执行 Agent 在 ReAct 循环每轮开始前检查信号
- 暂停粒度：**当前 ReAct 迭代完成后停下**（不会中断正在执行的工具调用，但不会再开始下一轮思考）
- 继续时从断点恢复（下一轮 ReAct 迭代开始）
- **WS 并发处理**：图谱执行期间，WS handler 必须能同时接收用户操作（暂停/继续/决策），不能被事件推送阻塞

### 9.2 审核重试与兜底

- 每个任务最多重试 `MAX_AGENT_RETRY`（当前 = 3）次
- 审核超限后，**不静默继续**，审核员（铁口）在群里发说明（LLM 生成），**在话术中自然引导用户做出选择**：

**交互方式**：
1. 铁口发群聊消息，用自然语言说明失败原因，并在话术中给出三个选项的引导（如"你可以让我跳过继续，也可以交还 @明哲 重新规划，或者干脆终止"）
2. 用户输入框区域显示**三个快捷回复按钮**：`跳过继续` / `交还明哲` / `终止执行`
3. 用户也可以**自由输入文本**，后端用意图识别判断用户选择（类似 `is_user_confirmation()` 逻辑）

| 选项 | 行为 |
|------|------|
| **跳过继续** | 记录失败，后续任务继续推进 |
| **交还明哲** | 架构师基于失败原因对该任务点重新规划（**单任务替换**：只替换失败任务，任务 ID 不变，插回 DAG 原位置重新执行），此选项每个任务只能用一次 |
| **终止执行** | 停止整个流水线 |

- **局部重规划**后的任务若仍超限，只剩"跳过继续"或"终止执行"，不再允许再次交还架构师（防止无限循环）

### 9.3 局部蓝图重规划

1. 铁口推送超限消息 → 用户选择"交还明哲"
2. 后端向明哲发送：失败任务的原始描述 + 执行结果 + 铁口的审核意见
3. 明哲生成**替代方案**（更换工具/更换思路），输出格式与原蓝图任务项一致
4. 替代方案**替换** DAG 中失败任务的节点（同一 task_id），依赖关系不变
5. 分配给同一个或新的执行 Agent 重新执行
6. 替代方案只有一次机会，若仍失败则只能跳过或终止

### 9.4 执行汇报策略

**汇报节点由架构师定义**：明哲在蓝图中为每个任务定义 `checkpoints`（汇报节点），如"完成文件创建后汇报"、"依赖安装完成后汇报"。执行 Agent 按照这些节点在群里汇报进度。

蓝图任务结构示例：
```json
{
  "id": "task_1",
  "name": "创建项目结构",
  "goal": "...",
  "checkpoints": ["项目目录创建完成", "配置文件写入完成"],
  "tools": ["shell", "python"]
}
```

**明哲任务分配消息**：蓝图确认后，明哲在群里发一条消息 @每个执行 Agent，说明分配的任务和汇报节点：

> @小辰 你负责「创建项目结构」，完成以下节点时请在群里汇报：1. 项目目录创建完成 2. 配置文件写入完成
> @小瑶 你负责「安装依赖」，完成后汇报一下

**汇报频率**：

| 节点 | 是否汇报 | 内容 |
|------|----------|------|
| 开始执行 | 是 | "收到任务「xxx」，开始干活！💪" |
| 工具调用过程 | **否** | 静默执行，不展示方法名和参数 |
| 架构师定义的 checkpoint | 是 | 按 checkpoint 描述汇报（LLM 生成自然语言） |
| 任务完成 | 是 | 最终结果总结 |

- 禁止在聊天气泡中展示工具调用的函数名、参数 JSON
- 最终汇报使用自然语言描述做了什么、产出了什么

### 9.5 同层并行执行

- 蓝图拓扑排序后，同层（无依赖关系的）任务**并行执行**
- 使用 `threading` 实现：同层任务分配到线程池，每个线程独立运行 ReAct 循环
- 消息通过 `emit_event`（线程安全）推送到 WS，前端看到的消息可能交错（不同 Agent 的发言交替出现），这是正常行为
- 跨层依赖仍然串行：第一层全部完成后才开始第二层

### 9.6 ReAct 迭代策略

- **无最大迭代上限**：不设 `MAX_ITERATIONS`，由 LLM 自行判断何时完成任务
- **安全退出**：如果 LLM 连续多轮既不调用工具也不产出新内容，自动判定为完成

## 12. WS 断线续跑策略

- **后端不停**：WS 断线后，后端流程继续执行，不因前端断开而暂停或终止
- **事件持久化**：所有事件通过 EventBus 推送时同步写入 `chat_history.json`，断线期间的事件不丢失
- **重连补发**：前端重连 WS 后，通过 REST API 拉取 `chat_history.json`，与本地已有事件做差集，补发断线期间的事件
- **状态同步**：重连后前端根据补发的事件恢复 `chatStore` 状态（当前阶段、活跃 Agent、是否在等待 HITL 等）
- **HITL 中断兼容**：若断线期间后端进入 HITL 等待（如蓝图确认、审核超限），重连后前端正常展示中断卡片，用户可继续操作

### 13.3 WebSocket action（客户端 → 服务端）

| action | 用途 |
|--------|------|

| `resume_pause` | 从暂停中恢复 |
| `decision` | 审核超限用户决策（skip/reroute/terminate） |
| `reconnect` | WS 断线重连 |

### 13.4 WebSocket 事件 type（服务端 → 客户端）

`phase_start`、`agent_join`、`agent_thinking`、`agent_output`、`hitl_interrupt`、`hitl_retry_limit`、`task_complete`、`error`

### 13.5 LangGraph 拓扑

节点顺序：`consultant` → `architect` → `hitl_blueprint` → `orchestrator`（调度多个小某执行） → `reviewer` → END
`interrupt_before`：`hitl_blueprint`
Checkpoint：`data/checkpoints.sqlite`

注意：
- 执行阶段没有统一的 "experts" 节点。orchestrator 直接调度多个执行 Agent（小某），每个任务独立创建身份、独立运行 ReAct 循环
- `hitl_blueprint` 是 HITL 中断点，不产生 `agent_join` 事件，只产生 `hitl_interrupt` 卡片
- `_iter_graph_events` 中 `consultant` 和 `architect` 节点的 `agent_join`/`agent_output` 已由 `_run_pipeline_after_confirm` 手动处理，图谱内部不再重复发送
